# 功能规格：IDEA Git 视图重构

**功能分支**: `012-idea-ui-refactor`
**创建日期**: 2026-04-04
**状态**: 草稿
**输入**: "重构现有的历史分支以及右键菜单，没有diff的文件全预览，标签贮藏和rebase也重构，使得更接近~/git/idea中的git功能和布局"

## 用户场景与测试 *(必填)*

### 用户故事 1 — 提交历史右键菜单重构 (优先级: P1)

用户在日志视图中右键点击某个提交时，需要看到与 IDEA 完全一致的操作菜单，包括：重置到此提交、还原提交、撤销提交(Uncommit)、修改提交消息(Reword)、Fixup 到某提交、Squash 到某提交、丢弃提交(Drop)、压缩多个提交(Squash Commits)、交互式变基、推送到此提交、从此提交新建分支、从此提交创建标签。菜单项应按照 IDEA 的分组和顺序排列。

**优先级原因**: 右键菜单是用户操作提交历史的核心入口，操作项不全或顺序不一致会直接影响日常工作效率。

**独立测试**: 打开日志视图，右键任意提交，验证菜单项完整且每个操作可正常执行。

**验收场景**:

1. **Given** 日志视图中选中一个提交, **When** 用户右键点击, **Then** 弹出菜单包含以下分组：历史重写操作（Reword、Fixup、Squash、Drop、Interactive Rebase）、提交操作（还原、撤销）、引用操作（新建分支、创建标签、推送到此提交）、重置操作（重置当前分支到此处）。
2. **Given** 选中任意提交（不限于 HEAD）, **When** 用户点击"撤销提交(Uncommit)", **Then** 从 HEAD 到该提交（含）的所有提交被撤销，改动合并返回暂存区，日志刷新。
3. **Given** 选中一个非 HEAD 提交, **When** 用户点击"修改提交消息(Reword)", **Then** 弹出消息编辑框，确认后通过交互式变基完成消息修改。
4. **Given** 选中多个连续提交, **When** 用户点击"压缩提交(Squash Commits)", **Then** 弹出消息编辑框合并消息，确认后压缩为单个提交。

### 用户故事 2 — 分支面板右键菜单重构 (优先级: P1)

用户在分支弹出窗口或日志侧边栏的分支面板中右键点击分支时，需要看到与 IDEA 一致的操作菜单：检出、新建分支、合并到当前、变基当前到此、比较、跟踪分支、拉取、推送、重命名、删除。远程分支组还应支持编辑和移除操作。

**优先级原因**: 分支操作是仅次于提交的高频操作，菜单缺失或布局不一致会降低用户切换效率。

**独立测试**: 在分支弹出窗口中右键本地分支和远程分支，验证菜单项完整且操作正确。

**验收场景**:

1. **Given** 分支弹出窗口已打开, **When** 右键一个本地分支, **Then** 菜单显示：检出、新建分支从此、合并到当前、变基当前到此、比较、跟踪分支、拉取、推送、重命名、删除。
2. **Given** 右键一个远程分支, **When** 查看菜单, **Then** 菜单包含：检出（创建本地跟踪分支）、新建分支从此、合并到当前、比较、拉取。
3. **Given** 选中当前分支（HEAD）, **When** 右键, **Then** "检出"和"删除"菜单项为禁用状态（灰色）。

### 用户故事 3 — 无 Diff 文件全文预览 (优先级: P1)

当用户在变更列表中选择一个新增文件（Untracked/Added）或选中一个没有差异的文件时，右侧面板应显示该文件的完整内容预览（带语法高亮），而非空白的 diff 视图或"无差异"提示。这与 IDEA 的 Staging Area 行为一致。

**优先级原因**: 新增文件是最常见的无 diff 场景，用户需要快速预览文件内容来确认是否要暂存。

**独立测试**: 创建一个新文件（未跟踪），在变更列表中选中它，验证右侧显示完整文件内容而非空白。

**验收场景**:

1. **Given** 仓库中有一个新建的未跟踪文件, **When** 用户在变更列表中选中该文件, **Then** 右侧 diff 面板显示该文件的完整内容，带语法高亮，所有行标记为"新增"（绿色）。
2. **Given** 一个已暂存但无修改的文件, **When** 用户选中该文件, **Then** 右侧显示文件完整内容而非"无差异"空白。
3. **Given** 一个二进制文件（如图片）, **When** 用户选中, **Then** 显示"二进制文件，无法预览"的友好提示，而非崩溃或空白。

### 用户故事 4 — 标签管理视图重构 (优先级: P2)

标签创建对话框和标签列表需要重构为与 IDEA 一致的布局：包含仓库选择器、当前分支显示、标签名输入框、强制覆盖选项、提交引用输入框（带验证按钮）、消息文本区域。标签列表需支持右键菜单：推送标签、删除本地标签、删除远程标签。

**优先级原因**: 标签管理用于发布流程，布局不一致会增加操作失误的风险。

**独立测试**: 打开标签对话框，创建一个带消息的标签，验证验证按钮可用，标签创建成功后在列表中右键可推送和删除。

**验收场景**:

1. **Given** 标签对话框已打开, **When** 用户查看布局, **Then** 依次显示：当前分支、标签名输入框、强制覆盖复选框、提交引用输入框+验证按钮、消息文本区域。
2. **Given** 用户输入提交引用"abc123", **When** 点击"验证"按钮, **Then** 系统验证该引用是否存在并显示结果。
3. **Given** 标签列表中选中一个标签, **When** 右键点击, **Then** 菜单显示：推送到远程、删除本地、删除远程、删除本地和远程。

### 用户故事 5 — 贮藏面板重构 (优先级: P2)

贮藏面板需要重构为与 IDEA 一致的布局和操作：贮藏创建对话框包含消息编辑器（多行文本输入框）和"保留暂存区(Keep Index)"选项。贮藏列表以树形结构按仓库分组显示，每个贮藏项显示摘要和时间戳。右键菜单包含：弹出(Pop)、应用(Apply)、应用到新分支(Unstash As)、丢弃(Drop)、清空所有(Clear)。

**优先级原因**: 贮藏是开发者频繁使用的临时存储工具，IDEA 的交互设计已被广泛验证。

**独立测试**: 创建一个贮藏，在列表中右键，验证所有操作可用且执行正确。

**验收场景**:

1. **Given** 工作区有未提交的改动, **When** 用户点击"贮藏", **Then** 弹出对话框包含：消息编辑器、"保留暂存区"复选框。
2. **Given** 贮藏列表中存在多个贮藏, **When** 右键某个贮藏, **Then** 菜单显示：弹出、应用、应用到新分支、丢弃、清空所有。
3. **Given** 用户选择"应用到新分支", **When** 输入新分支名, **Then** 系统从贮藏创建新分支并应用改动，贮藏保留在列表中。
4. **Given** 用户选择"清空所有", **When** 确认操作, **Then** 所有贮藏被删除。

---

### 用户故事 6 — 交互式变基编辑器重构 (优先级: P2)

交互式变基编辑器需要重构为与 IDEA 一致的布局：顶部工具栏（上移/下移按钮 + Pick/Edit 快捷按钮）、提交表格（操作列 + 哈希列 + 消息列）、底部提交详情面板（显示选中提交的完整信息和改动列表）。支持拖拽重排和行内消息编辑。右键菜单包含所有操作选项。

**优先级原因**: 交互式变基是高级但关键的工作流，IDEA 的可视化编辑器大幅降低了操作难度。

**独立测试**: 发起一次交互式变基，验证编辑器布局正确、拖拽重排可用、操作切换正常。

**验收场景**:

1. **Given** 用户发起交互式变基, **When** 编辑器打开, **Then** 显示：顶部工具栏（上移↑、下移↓、Pick、Edit 按钮）、提交表格（操作/哈希/消息三列）、底部选中提交详情。
2. **Given** 编辑器中选中一行, **When** 用户拖拽该行到新位置, **Then** 提交顺序更新。
3. **Given** 将某行操作切换为 Reword, **When** 双击消息列, **Then** 进入行内编辑模式，可修改提交消息。
4. **Given** 右键某行, **When** 查看菜单, **Then** 显示所有操作选项：Pick、Reword、Edit、Squash、Fixup、Drop。

### 边界情况

- 当提交历史为空（初始仓库无提交）时，右键菜单和日志视图应显示友好的空状态提示。
- 当选中多个不连续的提交进行 Squash 时，系统应提示"仅支持连续提交的压缩"。
- 当文件内容超过 1MB 或 5000 行时，全文预览应截断并提示"文件过大，仅显示前 5000 行"而非卡顿。
- 当贮藏应用产生冲突时，系统应自动进入冲突解决流程。
- 当交互式变基过程中遇到冲突时，编辑器应暂停并显示冲突文件列表和"继续/中止/跳过"控制按钮。
- 当远程推送标签失败（权限不足）时，应显示明确的错误信息。

### 功能需求

- **FR-001**: 提交历史右键菜单必须按 IDEA 分组顺序包含以下操作：重置到此、还原、撤销(Uncommit)、修改消息(Reword)、Fixup、Squash Into、Drop、Squash Commits（多选）、交互式变基、推送到此提交、新建分支、创建标签。
- **FR-002**: 分支右键菜单必须包含：检出、新建分支从此、合并到当前、变基当前到此、比较、跟踪分支、拉取、推送、重命名、删除。当前分支的"检出"和"删除"必须为禁用状态。
- **FR-003**: 选中无 diff 的文件（新增文件、未跟踪文件）时，diff 面板必须显示文件完整内容（带语法高亮），所有行标记为新增。超过 1MB 或 5000 行的文件截断显示并提示。
- **FR-004**: 选中二进制文件时，diff 面板必须显示"二进制文件，无法预览"提示，不得崩溃或显示乱码。
- **FR-005**: 标签创建对话框必须包含：当前分支显示、标签名输入、强制覆盖选项、提交引用输入+验证按钮、消息文本区域。
- **FR-006**: 标签列表右键菜单必须包含：推送到远程、删除本地、删除远程、同时删除本地和远程。
- **FR-007**: 贮藏创建对话框必须包含消息编辑器（多行文本输入框）和"保留暂存区(Keep Index)"选项。
- **FR-008**: 贮藏列表右键菜单必须包含：弹出(Pop)、应用(Apply)、应用到新分支(Unstash As)、丢弃(Drop)、清空所有(Clear)。
- **FR-009**: "应用到新分支"操作必须弹出分支名输入框，支持分支名验证，创建新分支并应用贮藏改动。
- **FR-010**: 交互式变基编辑器必须包含：顶部工具栏（上移/下移/Pick/Edit按钮）、三列提交表格（操作/哈希/消息）、底部提交详情面板。
- **FR-011**: 变基编辑器必须支持拖拽重排提交行。
- **FR-012**: 变基编辑器的消息列必须支持双击进入行内编辑模式。
- **FR-013**: 变基编辑器右键菜单必须包含所有操作选项：Pick、Reword、Edit、Squash、Fixup、Drop。
- **FR-014**: 所有重构后的视图必须保持现有的 MotionSites 深色主题（背景 #09090b，强调色 #415fff）。
- **FR-015**: 所有菜单项文本必须使用中文标签，与 IDEA 中文本地化一致。

### 关键实体

- **提交操作(Commit Action)**: 用户对选中提交执行的操作，包含操作类型（重置/还原/撤销/修改消息/Fixup/Squash/Drop）和目标提交。
- **分支操作(Branch Action)**: 用户对选中分支执行的操作，包含操作类型和是否需要确认。
- **文件预览(File Preview)**: 当文件无差异时显示的完整文件内容，包含文件路径、内容、语法高亮类型、是否为二进制。
- **贮藏条目(Stash Entry)**: 贮藏的快照信息，包含索引、消息、来源分支、时间戳、是否含未跟踪文件。

## 假设

- 011-idea-git-parity 分支的代码已合并，所有新 git-core 模块（blame、signature、worktree、submodule、graph）可用。
- IDEA 的 ~/git/idea 源码作为参考实现，不做逐行复制，而是匹配功能和布局模式。
- 现有 MotionSites 深色主题不变，仅调整视图布局和菜单结构。
- 中文（简体）仍为主要界面语言。
- 本次重构主要重组现有 UI 组件和补充缺失的菜单项，需少量 git-core 扩展：
  - stash_save_with_options 新增 keep_index 参数（git stash push --keep-index）。
  - 新增"应用到新分支(Unstash As)"操作：从贮藏创建分支并应用。
  - 新增 uncommit 操作：软重置从 HEAD 到指定提交。
