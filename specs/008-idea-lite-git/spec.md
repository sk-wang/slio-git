# Feature Specification: IDEA 式 Git 工作台主线

**Feature Branch**: `008-idea-lite-git`  
**Created**: 2026-03-25  
**Status**: Draft  
**Input**: User description: "这个软件的发展能成为一个阉割代码功能的 idea，让我能快速查看变更和操作 git"

## Product Vision

把 `slio-git` 发展成一款“去掉代码编辑能力、只保留 Git 主工作流”的 IDEA 式桌面工具：用户打开仓库后，能够立刻看懂当前上下文，快速审阅改动，并在同一个工作台中完成大多数日常 Git 操作，而不是在终端、零散弹窗和多个小工具之间来回切换。

## Experience Principles

- **先看懂，再操作**：用户先快速建立“当前仓库 / 当前分支 / 当前状态 / 当前风险”的认知，再进入 Git 动作
- **主线闭环优先**：优先保障“看变更 → 暂存 → 提交 → 同步 → 切分支”这条最高频闭环，而不是先堆低频能力
- **风险状态留在站内处理**：冲突、同步异常、认证问题和进行中的 Git 流程应尽量在应用内被看懂并继续处理
- **像 IDEA，但不变成 IDE**：保留 IDEA/PhpStorm 式工作台体验、上下文连续性和信息层次，但不扩展为代码编辑器

## In Scope

- 围绕当前仓库的主工作台体验
- 变更审阅、差异预览与状态判断
- 高频 Git 操作闭环：暂存、提交、抓取、拉取、推送、分支切换
- 最近项目、上下文恢复与多仓库切换
- 冲突、同步异常、认证失败等风险状态的引导处理
- 历史、标签、储藏等用于支持主工作流判断的次级 Git 上下文

## Out of Scope

- 代码编辑、重构、补全、跳转、调试、运行配置等 IDE 编码能力
- 插件市场、通用项目管理或与 Git 主流程无关的桌面壳层能力
- 取代终端的全部 Git 命令覆盖；目标是覆盖高频主流程，而不是所有低频边角命令
- 把产品做成完整协作平台、代码评审平台或 DevOps 门户

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 打开后立刻进入 Git 工作台 (Priority: P1)

作为用户，我希望这个软件像一个去掉写代码能力的 IDEA：打开仓库后，第一眼就能看到当前仓库、当前分支、变更列表、差异预览和同步状态，不需要在多个页面里找入口。

**Why this priority**: 如果第一屏不能快速建立“我现在在哪个仓库、改了什么、能做什么”的认知，这个产品就无法承担日常 Git 主工作台的角色。

**Independent Test**: 打开一个有未提交改动的仓库后，用户应能在单一主工作台中快速识别当前分支、变更类型、改动内容和下一步可执行动作。

**Acceptance Scenarios**:

1. **Given** 用户打开了一个本地 Git 仓库，**When** 主界面完成加载，**Then** 系统会在单一主工作台中展示仓库上下文、当前分支、变更列表和改动预览
2. **Given** 当前仓库包含未暂存、已暂存或冲突中的文件，**When** 用户查看主工作台，**Then** 系统会清楚区分这些状态并帮助用户快速定位重点文件
3. **Given** 用户在变更列表中切换文件，**When** 选择不同改动项，**Then** 差异预览会立即跟随切换，帮助用户快速审阅改动

---

### User Story 2 - 在一个地方完成日常 Git 操作闭环 (Priority: P1)

作为用户，我希望在同一个工作台里完成日常 Git 主流程，包括审阅改动、暂存与取消暂存、提交、拉取、推送、抓取和切换分支，而不是频繁跳回终端或其他 Git 工具。

**Why this priority**: “快速查看变更和操作 Git”是产品的核心承诺；如果高频流程仍需要离开应用，就无法成为日常主工具。

**Independent Test**: 选取一个普通开发日常场景，从查看改动到完成一次提交并同步到远端，应能全程在应用内完成，不依赖终端。

**Acceptance Scenarios**:

1. **Given** 用户已经打开带有本地改动的仓库，**When** 用户准备提交变更，**Then** 系统会允许其在当前工作台中完成暂存、填写提交说明和创建提交
2. **Given** 当前分支已关联远端，**When** 用户执行抓取、拉取或推送，**Then** 系统会在同一工作流中完成操作并反馈成功、失败或需要处理的后续动作
3. **Given** 用户需要切换到另一个分支继续工作，**When** 用户打开分支入口，**Then** 系统会提供足够快且足够清晰的分支浏览、搜索和切换体验

---

### User Story 3 - 像 IDE 一样连续切换项目和恢复上下文 (Priority: P2)

作为用户，我希望这个软件具备 IDE 式的工作连续性：记住最近打开的仓库、恢复上次工作上下文，并允许我在多个项目之间快速切换，而不是每次都重新从零开始。

**Why this priority**: 一个真正能替代“终端 + 零散 Git 小工具”的日常桌面主工具，必须支持持续工作，而不是只适合一次性操作。

**Independent Test**: 打开多个仓库后关闭应用，再次启动时，用户应能迅速回到上次工作仓库，并从最近项目中切换到其他仓库继续工作。

**Acceptance Scenarios**:

1. **Given** 用户此前已经成功打开过多个仓库，**When** 重新启动应用，**Then** 系统会恢复最近的工作上下文并展示最近项目列表
2. **Given** 用户需要切换到另一个最近项目，**When** 在项目列表中选择目标仓库，**Then** 系统会快速切换到该仓库的主工作台
3. **Given** 最近项目中的某个路径已失效，**When** 用户重新打开应用或尝试切换，**Then** 系统会安全处理无效记录而不阻断其余项目使用

---

### User Story 4 - 遇到风险状态时也能留在同一工作台处理 (Priority: P2)

作为用户，我希望当仓库进入冲突、远端落后/领先、认证失败、变基中或其他异常状态时，系统给我的不是冰冷报错，而是清晰的状态解释和下一步操作，让我仍能把它当作日常 Git 工作台继续使用。

**Why this priority**: 一款只在“顺利时可用”的 Git 工具无法承担主工具角色；真正的替代价值来自风险状态下也不逼用户退回终端。

**Independent Test**: 制造一个包含冲突或远端同步问题的仓库，用户应能在应用中看懂当前状态、继续处理问题，并明确知道操作结果。

**Acceptance Scenarios**:

1. **Given** 仓库存在冲突或合并中断，**When** 用户进入主工作台，**Then** 系统会清楚提示当前风险状态并提供进入处理流程的入口
2. **Given** 用户执行远端操作失败，**When** 操作返回结果，**Then** 系统会说明失败原因、当前影响范围和建议的下一步动作
3. **Given** 用户完成冲突解决或其他风险处理，**When** 回到主工作台，**Then** 系统会同步更新仓库状态并确认当前是否已恢复正常工作流

---

### User Story 5 - 需要回看历史时也不必离开应用 (Priority: P3)

作为用户，我希望在做提交、同步、切分支或处理异常前，可以方便地回看最近提交、标签、储藏等 Git 上下文，这样我在判断下一步动作时不需要回终端查历史。

**Why this priority**: 这不是最核心的第一步，但如果缺少历史上下文，很多 Git 操作会变成“盲做”，产品仍然难以成为日常主工具。

**Independent Test**: 在一个已有多次提交记录的仓库中，用户应能在应用内回看近期提交与关键上下文，再继续当前 Git 操作。

**Acceptance Scenarios**:

1. **Given** 用户准备提交、拉取、推送或切换分支，**When** 需要确认最近仓库历史，**Then** 系统会提供便捷的历史上下文查看入口
2. **Given** 用户需要理解当前仓库为什么处于某种状态，**When** 查看近期提交、标签或储藏信息，**Then** 系统会帮助其建立更完整的判断依据
3. **Given** 用户查看完历史上下文，**When** 返回主工作台，**Then** 当前变更审阅和 Git 主操作路径仍保持连续，不被打断

### Edge Cases

- 当仓库改动很多、包含长文件名、二进制文件或大量目录层级时，主工作台仍需保持可浏览和可定位
- 当仓库处于 detached HEAD、merge、rebase、cherry-pick 或 revert 等特殊 Git 状态时，系统至少需要明确说明当前上下文；本轮优先覆盖 detached HEAD、merge/rebase 中与远端失败场景
- 当当前分支没有关联远端、远端不可达或认证失败时，系统需要给出可理解的结果，而不是只显示底层错误
- 当应用打开期间仓库被外部工具修改时，系统需要及时反映状态变化，避免让用户基于过期信息操作
- 当最近项目记录已失效或仓库目录被移动时，系统需要安全降级，不得导致启动或切换流程卡住
- 当仓库包含子模块、二进制文件或无法直接预览的内容时，系统需要说明限制并保持主工作流可继续
- 当提交历史很长或分支很多时，系统需要帮助用户快速定位上下文，而不是让历史入口变成新的复杂管理页

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 将“查看变更并执行 Git 操作”作为产品的一等主线，而不是把主要入口分散到多个互不连贯的页面
- **FR-002**: 系统 MUST 在打开仓库后提供一个稳定的主工作台，集中展示仓库身份、当前分支、变更状态、差异预览和同步上下文
- **FR-003**: 用户 MUST 能够从变更列表快速定位文件，并在切换文件时立即看到对应改动内容
- **FR-004**: 用户 MUST 能够在主工作流中完成暂存、取消暂存、放弃改动和提交前审阅，不必离开应用
- **FR-005**: 用户 MUST 能够在应用内完成抓取、拉取、推送等高频远端操作，并看到清晰结果反馈
- **FR-006**: 用户 MUST 能够浏览、搜索、切换和创建分支，并清楚识别当前分支及其远端关系
- **FR-007**: 系统 MUST 为冲突、合并中断、变基中和远端同步异常等高风险状态提供明确提示和继续处理入口
- **FR-008**: 系统 MUST 允许用户在应用内完成常见文本冲突的查看、决策与解决，而不是默认要求外部工具
- **FR-009**: 系统 MUST 记住最近使用的仓库和最近一次工作上下文，以支持重新启动后的快速恢复
- **FR-010**: 用户 MUST 能够在多个最近项目之间快速切换，并保持“进入即工作”的连续体验
- **FR-011**: 系统 MUST 在每次会改变仓库状态的操作后反馈成功、失败或待处理状态，避免用户需要猜测操作是否生效
- **FR-012**: 系统 MUST 在无改动、无上游分支、无远端或空仓库等场景下，继续保持可理解且可继续操作的工作台
- **FR-013**: 系统 MUST 保持 Git 主流程的高可达性，使用户完成日常操作时通常不需要打开终端
- **FR-014**: 系统 MUST 以“像一个只做 Git 的 IDEA”为产品方向，但不得扩展为完整代码编辑器、插件平台或通用 IDE
- **FR-015**: 系统 MUST 保持围绕当前仓库与当前工作上下文的专注体验，避免被低频管理功能抢走主视觉和主路径
- **FR-016**: 系统 MUST 让历史、标签、储藏、远端管理等次级 Git 能力保持可发现、可进入，但不得压过变更审阅和高频操作主线
- **FR-017**: 系统 MUST 在用户返回应用时尽快恢复到“可审阅、可判断、可操作”的状态，而不是要求重复初始化工作上下文
- **FR-018**: 系统 MUST 为近期提交、标签、储藏等历史上下文提供快速查看入口，以支撑提交、同步和分支操作前的判断
- **FR-019**: 系统 MUST 明确区分主工作流与辅助能力，确保用户第一眼看到的是变更审阅与高频 Git 操作，辅助能力只在需要时进入
- **FR-020**: 系统 MUST 持续保持“Git-first、面向单个开发者日常高频使用”的产品定位，不得偏离到完整代码编辑器、通用 IDE 或低频专家命令中心
- **FR-021**: 系统 MUST 在用户需要临时查看辅助上下文后，仍能快速回到当前仓库的主工作台，不让辅助能力打断主流程节奏

### Key Entities *(include if feature involves data)*

- **Git Workspace**: 用户进入仓库后看到的主工作台，承载仓库上下文、变更审阅与高频 Git 操作
- **Repository Context**: 当前仓库身份、当前分支、同步关系和风险状态等组成的工作背景信息
- **Change Review Surface**: 变更列表与差异预览组合而成的审阅区域，用于帮助用户快速理解改动
- **Git Action Flow**: 用户从审阅改动到暂存、提交、同步、切换分支的一系列连续操作路径
- **Project Memory**: 最近项目与最近工作上下文，帮助用户跨会话恢复和切换仓库
- **Risk State**: 冲突、同步失败、认证问题、变基中等会打断正常工作流但需要被引导处理的状态

## Assumptions

- 产品目标是成为“以 Git 工作流为核心的 IDEA 式桌面工具”，而不是全功能 IDE
- 代码编辑、语言服务、调试、运行配置和插件生态不在本次产品方向主线范围内
- 用户的主要目标是快速了解当前仓库状态并完成高频 Git 操作，而不是进行复杂仓库管理或团队治理
- 当前已经存在的一些项目记忆、差异查看、冲突处理和远端交互能力可以作为这一新 spec 的基础能力继续演进
- 主工作台应优先服务单个用户的日常连续工作体验，再逐步扩展低频能力的深度

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从打开应用或恢复会话开始，应能在 10 秒内识别当前仓库、当前分支、变更概况和下一步可操作入口
- **SC-002**: 对于一个典型的小到中等改动集，用户应能在 2 分钟内完成“审阅改动 → 暂存/取消暂存 → 提交 → 推送”的完整闭环，而不依赖终端
- **SC-003**: 用户从最近项目切换到另一仓库后，应能在 5 秒内重新进入该仓库的主工作台并开始审阅变更
- **SC-004**: 至少 90% 的日常 Git 高频任务应可直接在应用内完成，包括查看改动、暂存、提交、抓取、拉取、推送、分支切换和常见文本冲突处理
- **SC-005**: 在冲突、同步失败、认证失败或其他风险状态下，系统每次都应提供明确状态解释与下一步动作，避免仅留下原始报错
- **SC-006**: 新用户在首次使用时，应能在 30 秒内理解这是一款“围绕 Git 工作流的精简 IDEA 式工具”，而不是普通设置面板或零散 Git 对话框集合
- **SC-007**: 对于重复使用者，大多数工作会话应能够直接从最近仓库或已恢复上下文开始，而不是重新选择项目和重新建立状态认知
- **SC-008**: 用户在需要查历史、标签或储藏信息时，应能在 15 秒内找到相关上下文并回到当前 Git 主流程
- **SC-009**: 用户在需要切换分支时，应能在 10 秒内定位当前分支、搜索目标分支并发起切换
