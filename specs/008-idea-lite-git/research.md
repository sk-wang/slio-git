# Research: IDEA 式 Git 工作台主线

## 1. 产品中心应该是“单一 Git 工作台”而不是功能目录

- **Decision**: 打开仓库后以一个稳定的 Git 工作台作为绝对中心，围绕当前仓库、当前分支、变更审阅和高频 Git 操作组织所有主路径。
- **Rationale**: 用户的核心诉求不是“这个软件要有什么功能菜单”，而是“像 IDEA 一样，打开就能立刻看懂并开始做 Git”。这要求上下文集中、焦点稳定，而不是功能分散。
- **Alternatives considered**:
  - 按“提交 / 分支 / 远端 / 历史”拆成平级页面：入口清楚，但会不断打断仓库上下文
  - 保持当前多对话框拼接式结构：短期实现成本低，但难以形成主工具心智

## 2. 高价值闭环是“看变更 → 暂存 → 提交 → 同步”

- **Decision**: 优先把改动审阅和高频 Git 闭环定义为主线，其他能力围绕这条主线补位。
- **Rationale**: 绝大多数日常使用都围绕当前工作副本展开；如果这条链路足够顺，用户才会自然把应用作为默认 Git 工作台。
- **Alternatives considered**:
  - 先追求所有 Git 命令覆盖：范围太大，容易牺牲高频体验
  - 先做大量仓库管理能力：不符合“快速看变更和操作 Git”的核心诉求

## 3. 辅助上下文应该是“peek surface”，不是新的主战场

- **Decision**: 历史、标签、储藏、远端等能力保持可发现，但默认作为帮助判断的辅助视图或临时工作面存在，并允许快速返回主工作台。
- **Rationale**: 这些能力确实重要，但它们的主要价值是帮助用户理解当前状态与下一步动作，而不是替代变更审阅成为首屏焦点。
- **Alternatives considered**:
  - 把每类能力都提升为主导航：信息架构会重新变重，用户一进入产品就失焦
  - 把次级能力彻底隐藏：会导致很多判断仍需回终端

## 4. 风险状态必须被视为主工作流的一部分

- **Decision**: 冲突、认证失败、远端异常、rebase/merge 中断等状态，不视为“异常边角”，而是主工作流中必须被引导处理的正式场景。
- **Rationale**: Git 工具是否能成为主工具，关键不在顺利时，而在出问题时用户是否还能留在站内继续完成任务。
- **Alternatives considered**:
  - 失败时仅给底层报错：用户看不懂，也不知道下一步干什么
  - 默认跳外部 merge 工具或终端：破坏工作连续性

## 5. 最近项目与上下文恢复是 IDE 感的重要来源

- **Decision**: 最近项目、上次仓库恢复和多仓库快速切换继续作为核心能力，而不是“辅助便利功能”。
- **Rationale**: 用户要的是“IDEA 式连续工作感”，这意味着每次回到应用都不应重新建立上下文。
- **Alternatives considered**:
  - 每次手工重新打开仓库：简单，但无法形成高频主工具体验
  - 只记住仓库，不记住工作上下文：恢复速度仍然不够理想

## 6. “像 IDEA”指向 Git 工作方式，而不是完整 IDE 范围

- **Decision**: 产品方向明确为“Git-first 的 IDEA Lite”，保留 IDEA/PhpStorm 式工作台体验、上下文连续性和交互语义，但不向代码编辑器演进。
- **Rationale**: 这能帮助后续所有设计判断保持一致：什么时候要做得更像 IDE，什么时候必须主动止步。
- **Alternatives considered**:
  - 模糊地说“像 IDE”：容易让范围不断膨胀
  - 完全不借鉴 IDEA 心智：会失去用户最看重的体验锚点

## 7. 继续以 IntelliJ-compatible 为 Git 语义底线

- **Decision**: 即使产品范围聚焦在 Git 工作台，Git 交互算法、用户流和状态语义仍以 IntelliJ/PhpStorm 为基准。
- **Rationale**: Constitution 已将 IntelliJ compatibility 设为最高约束；而且这正是用户选择本项目的重要原因。
- **Alternatives considered**:
  - 为了更快交付而自行简化 Git 语义：会直接损害核心产品信任
  - 完全照搬完整 IDE 壳层：超出本产品范围，也不必要
