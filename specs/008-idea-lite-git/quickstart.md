# Quickstart: IDEA 式 Git 工作台主线

## 1. Build & Verification

在实现完成后执行：

```bash
cargo check -p src-ui
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
```

## 2. Launch the UI

```bash
cargo run -p src-ui
```

启动后优先确认：

- 仓库打开后默认进入的是 Git 主工作台，而不是管理型首页
- 中文字体、中文文案和状态提示保持正常
- 第一眼能看到仓库、分支、改动与下一步动作

## 3. Core Workspace Walkthrough

1. 打开一个存在未提交改动的仓库。
2. 确认首屏同时具备：
   - 当前仓库上下文
   - 当前分支与同步提示
   - 变更列表
   - 当前选中文件的差异预览
3. 在变更列表中切换多个文件，确认 diff 预览立即跟随变化。
4. 人工判断第一屏是否把“看变更 + 做 Git”作为绝对主线，而不是把历史、远端、标签等管理入口放到主视觉中心。

## 4. High-Frequency Git Loop Walkthrough

1. 在主工作台中选择部分文件并执行暂存 / 取消暂存。
2. 打开提交入口并填写中文提交说明。
3. 完成一次提交。
4. 对当前分支执行抓取、拉取或推送。
5. 确认整个过程无需离开应用或依赖终端。
6. 每一步都应看到明确成功、失败或待处理反馈。

## 5. Branch and Context Walkthrough

1. 从主工作台打开分支入口。
2. 验证能够：
   - 看清当前分支
   - 搜索分支
   - 切换分支
   - 理解与远端的关系
3. 打开历史、标签和储藏视图，确认都能查看当前上下文并返回主工作台。
4. 若仓库中存在 stash，确认至少可以看到 stash 列表并触发一项 stash 相关动作的入口。
5. 确认返回后仍然保持当前仓库主流程，不需要重新建立认知。

## 6. Continuity Walkthrough

1. 打开两个不同仓库。
2. 关闭应用并重新启动。
3. 确认：
   - 上次仓库能恢复
   - 最近项目可见
   - 切换仓库后能快速进入对应主工作台
4. 删除最近项目中的一个路径后再次验证，确认无效记录不会阻断其他项目使用。

## 7. Risk-State Walkthrough

1. 准备以下任一场景：
   - 存在冲突的仓库
   - detached HEAD 的仓库
   - 当前分支远端被拒绝推送
   - 认证失败的远端操作
   - 进行中的 rebase / merge
2. 进入应用后确认系统不会只显示底层报错，而是会：
   - 说明当前状态
   - 告知影响范围
   - 提供下一步动作或处理入口
3. 完成处理后返回主工作台，确认仓库状态同步更新。

## 8. Acceptance Mapping

| Success Criteria | Verification Focus |
|------------------|--------------------|
| SC-001 | 打开仓库后 10 秒内是否能识别仓库、分支、变更概况和下一步动作 |
| SC-002 | 能否在应用内完成“审阅改动 → 暂存/取消暂存 → 提交 → 推送”闭环 |
| SC-003 | 最近项目切换后是否能在 5 秒内回到目标仓库主工作台 |
| SC-004 | 高频 Git 任务是否都能在应用内完成，不被迫回终端 |
| SC-005 | 风险状态是否有解释、有入口、有后续动作 |
| SC-006 | 新用户是否能快速理解这是 Git-first 的 IDEA Lite，而不是设置页集合 |
| SC-007 | 重复使用者是否主要从恢复上下文或最近仓库开始工作 |
| SC-008 | 查看历史/标签/储藏后能否快速回到当前 Git 主流程 |
| SC-009 | 能否在 10 秒内定位当前分支、搜索目标分支并发起切换 |

## 9. Performance Validation

1. 冷启动应用并记录到主窗口可交互的大致耗时，确认是否接近 `<300ms` 的目标。
2. 在中小型仓库中执行以下高频操作并观察日志或人工体感延迟：
   - 打开仓库并展示状态
   - 选择文件查看 diff
   - 打开历史视图
   - 执行一次抓取或状态刷新
3. 确认上述常见 Git 操作的感知延迟接近 `<100ms` 目标，若明显超出则记录触发场景与日志。
