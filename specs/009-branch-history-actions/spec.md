# Feature Specification: 分支视图提交动作补齐

**Feature Branch**: `009-branch-history-actions`  
**Created**: 2026-03-25  
**Status**: Updated  
**Input**: User description: "创建一个spec 实现这些在分支视图上的功能"（以用户提供的 PhpStorm / IDEA 分支视图截图为目标）

## Product Vision

把分支视图从“只够切分支的侧边入口”升级为一个接近 PhpStorm / IDEA 的轻量提交工作台：用户在查看某个分支的提交历史时，可以直接围绕某一条提交完成判断、比较、派生、发布和历史调整，而不是再切到终端、单独的历史页面或外部 Git 工具。

## Experience Principles

- **提交就是操作入口**：当用户已经看到某条提交时，常用动作应直接围绕该提交出现，而不是要求重新寻找别的入口
- **像 PhpStorm，但保持轻量**：动作分组、命名、可用性判断和风险提示尽量贴近 PhpStorm / IDEA 的分支与日志体验，但不把分支视图扩展成完整 IDE
- **危险动作近但不误触**：重写历史、移动分支指针、批量发布等高风险动作应保持高可达，同时必须在真正执行前讲清影响范围
- **状态感知优先**：是否可执行，取决于当前分支、上游关系、提交归属、工作区状态和进行中的 Git 流程；界面必须把这些前置条件说清楚
- **上下文不丢失**：完成复制、比较、建分支、回退等动作后，用户应尽量留在当前分支视图，继续沿着同一上下文操作
- **列表本身也要能说话**：当前分支、已选中分支、已打开菜单的分支，以及可折叠文件夹，应当只靠列表行本身就能被快速分辨
- **同类菜单同一语言**：分支菜单、提交菜单，以及相关历史视图中的基础提交菜单，应尽量共享同一种 JetBrains 式弹层节奏、分组和危险动作语义
- **滚动条不抢戏**：分支视图里的时间线、列表、比较区和其他内嵌滚动区域，要使用接近 PhpStorm 的低存在感滚动条，不得遮挡正文，也不能出现重复叠加的横向滚动条
- **新文件选中就能看**：当用户选中未跟踪文件或新增文件时，右侧预览区应直接显示文件内容或可理解的替代预览，而不是误报“没有变更”

## In Scope

- 分支视图内的提交列表、提交选中态与提交级上下文菜单
- 分支导航列表中的当前态、选中态、右键菜单态、文件夹折叠态与紧凑上下文按钮
- 分支级上下文菜单的分组、危险动作区与中文禁用原因
- 围绕单条提交的常用动作分组、可用性判断与中文提示
- 从提交派生新分支、新标签，或将当前工作上下文对齐到该提交
- 围绕当前分支的提交应用、撤销、重置、发布到指定位置等操作
- 对本地未发布提交的常见历史整理动作，如改说明、合并、丢弃、从此处开始整理
- 父提交 / 子提交导航，以及必要的“定位回当前上下文”能力
- 动作执行后的成功、失败、冲突与待继续状态反馈
- 历史视图中的基础提交右键菜单（至少覆盖复制哈希、导出补丁与查看详情），并保持与分支视图相近的菜单语言
- 分支视图内提交时间线、详情区、比较区等滚动容器的滚动条表现与溢出策略
- 与当前分支 / 历史上下文相关的文件预览区对新增文件、未跟踪文件的直接内容预览

## Out of Scope

- 完整替代专业 Git 日志工具的所有低频菜单项
- 任意脚本化 Git 命令输入、原始命令行编辑或通用终端能力
- 代码编辑、逐文件内容修改或超出 Git 操作本身的 IDE 能力
- 多仓库联动日志、团队评审面板或托管平台专属分支保护策略配置
- 完整自由编辑的历史整理剧本；本轮聚焦高频、可被明确引导的提交级动作

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在分支视图里直接对提交发起常用动作 (Priority: P1)

作为用户，我希望在分支视图中点中某条提交后，就能直接看到与这条提交相关的动作，比如复制版本号、创建补丁、对比、签出或基于它创建分支，而不是再跳去另一个页面或终端。

**Why this priority**: 这是用户从截图里直接期待的核心能力；如果分支视图仍然只能“看”不能“动”，它就无法承担 PhpStorm 式的日常 Git 操作入口。

**Independent Test**: 在一个包含多个提交的仓库中打开分支视图，选中任意可见提交并打开动作菜单，应能直接完成至少一种常用操作（例如复制哈希、建分支、创建补丁或发起比较），且不需要离开分支视图。

**Acceptance Scenarios**:

1. **Given** 用户在分支视图中选中了一条提交，**When** 打开该提交的上下文菜单，**Then** 系统会以稳定分组展示与该提交相关的常用动作，并使用中文标签与简短说明
2. **Given** 用户需要把这条提交的版本号发给别人，**When** 选择复制版本号，**Then** 系统会立即完成复制，并反馈复制成功而不改变当前视图上下文
3. **Given** 用户希望基于该提交创建新分支或新标签，**When** 选择相应动作并确认名称，**Then** 系统会以该提交作为起点完成派生并刷新分支视图状态

---

### User Story 2 - 从某条提交快速比较、定位和切换工作上下文 (Priority: P1)

作为用户，我希望在分支视图里直接以某条提交为参照进行比较、查看其上下游关系，或者切换到这条提交对应的工作上下文，这样我在判断“接下来该去哪、该基于谁继续工作”时不需要跳出当前视图。

**Why this priority**: 分支视图不仅是动作菜单，也是用户建立历史判断的地方；如果无法从提交快速对比和导航，就很难像 PhpStorm 一样形成连续决策。

**Independent Test**: 选中一条非 HEAD 提交后，用户应能在分支视图中发起与当前本地上下文的比较、跳转到父或子提交，或将工作上下文切换到该提交，而无需先进入独立历史页面。

**Acceptance Scenarios**:

1. **Given** 用户选中了历史中的某条提交，**When** 选择与当前本地上下文比较，**Then** 系统会以该提交为基准展示差异，并保留用户返回分支视图继续操作的路径
2. **Given** 用户正在追查某条提交的前后关系，**When** 选择前往父提交或子提交，**Then** 分支视图会更新选中状态并把焦点移动到对应提交
3. **Given** 用户希望临时切换到某条提交继续查看仓库状态，**When** 选择签出或类似动作，**Then** 系统会在执行前说明当前会话上下文将如何变化，并在完成后刷新当前状态

---

### User Story 3 - 在分支视图中处理“应用、撤销、重置、发布到这里”这类高价值操作 (Priority: P2)

作为用户，我希望当我看着某条提交时，能直接决定把它摘过来、撤销掉、把当前分支重置到这里，或者只把当前分支发布到这个位置，而不是记住一串 Git 命令再自己操作。

**Why this priority**: 这些动作频率没有复制或建分支那么高，但一旦需要，用户往往已经在看历史；把它们留在分支视图里，能显著减少“看懂后又要切走执行”的割裂感。

**Independent Test**: 在一个有本地提交、远端上游和可构造冲突的测试仓库中，用户应能从选中的提交发起摘取、撤销、重置或“推送到这里”，并获得明确的影响说明和结果反馈。

**Acceptance Scenarios**:

1. **Given** 用户在当前分支上看到另一条可应用的提交，**When** 选择摘取该提交，**Then** 系统会发起应用流程，并在成功、冲突或失败时明确告知下一步
2. **Given** 用户想把当前分支指针移动到选中的历史位置，**When** 选择重置到这里，**Then** 系统会在真正执行前说明对当前分支、工作区和未发布提交的影响
3. **Given** 当前分支配置了上游且选中的提交位于可发布范围内，**When** 选择推送到这里，**Then** 系统会仅针对当前分支的上游执行该发布动作，并在完成后刷新领先/落后状态

---

### User Story 4 - 对本地未发布提交进行 PhpStorm 式整理 (Priority: P2)

作为用户，我希望在分支视图里直接对本地未发布提交做常见整理，比如修改提交说明、与前一条合并、压缩、丢弃、从这里开始整理，甚至撤销最近一次提交，而不是为了这些“小手术”退回命令行。

**Why this priority**: 这类动作最能体现“像 IDEA 一样操作 Git”的深度，也是分支视图从“查看器”升级为“工作台”的关键差异点。

**Independent Test**: 在当前分支包含数条尚未发布的本地提交时，用户应能从分支视图对其中的合格提交发起改说明、合并、压缩、丢弃、撤销提交或从此处开始整理，并在不满足条件时看到明确禁用原因。

**Acceptance Scenarios**:

1. **Given** 当前分支存在本地未发布提交，**When** 用户对其中一条提交打开菜单，**Then** 系统会只对符合条件的提交显示或启用历史整理动作
2. **Given** 用户想把一条修补型提交并入前面的提交，**When** 选择合并、压缩或修补类动作，**Then** 系统会在执行前说明影响范围，并在执行后刷新提交序列
3. **Given** 历史整理过程中出现冲突或需要继续的流程，**When** 操作被中断，**Then** 系统会保留当前上下文并提供继续、跳过、取消或返回安全状态的入口

---

### User Story 5 - 分支列表与分支菜单也保持 PhpStorm 式的列表节奏 (Priority: P2)

作为用户，我希望左侧分支列表里的当前分支、已选中分支和正在打开菜单的分支能被明显区分，而且右侧上下文按钮和右键菜单都足够顺手，这样我不需要离开列表就能快速切换和管理分支。

**Why this priority**: 当分支视图真正变成工作台时，左侧分支导航不再只是“入口目录”，而是高频操作面；如果列表态和分支菜单太弱，整体就仍然不像 PhpStorm。

**Independent Test**: 在一个包含本地分支、远程分支和分组文件夹的仓库里打开分支视图，用户应能只靠列表行状态辨认当前分支、选中分支和菜单打开分支，并可通过右键或行尾按钮打开分支菜单完成至少一种分支动作。

**Acceptance Scenarios**:

1. **Given** 用户在分支列表中浏览本地和远程分支，**When** 当前分支、选中分支或菜单打开分支出现，**Then** 列表行会用稳定且不过度喧宾夺主的方式区分这些状态
2. **Given** 用户需要对某个分支执行签出、比较、推送或删除，**When** 右键该分支或点击行尾上下文按钮，**Then** 系统会打开与提交菜单风格一致的分支动作弹层，并按高频到危险动作的顺序分组
3. **Given** 用户在分支树中展开或折叠一个文件夹节点，**When** 节点切换状态，**Then** 文件夹行本身会反馈当前展开态，并保持与分支行一致的紧凑列表节奏

---

### User Story 6 - 历史视图中的提交也能右键完成基础动作 (Priority: P3)

作为用户，我希望在独立历史视图里右键某条提交时，也能立刻做基础动作，比如复制哈希或导出补丁，而不是只有在分支视图里才有这类能力。

**Why this priority**: 这不是分支工作台的第一优先级，但如果历史视图完全没有上下文动作，会破坏“同类菜单同一语言”的整体体验。

**Independent Test**: 打开历史视图并右键任意一条提交，应能看到基础提交菜单，并在不离开历史视图上下文的情况下完成复制哈希或导出补丁。

**Acceptance Scenarios**:

1. **Given** 用户在历史视图里查看提交列表，**When** 右键某条提交，**Then** 系统会打开一个紧凑的基础提交菜单，而不是毫无反馈
2. **Given** 用户想从历史视图中快速复制某条提交的哈希，**When** 选择复制哈希，**Then** 系统会直接完成复制并保留当前历史列表上下文
3. **Given** 用户想从历史视图中导出某条提交为补丁，**When** 选择导出补丁，**Then** 系统会完成导出并继续留在历史视图里

### Edge Cases

- 选中的提交是根提交、合并提交或有多个父提交时，系统必须清楚说明哪些动作可用、需要进一步选择目标，或为何不可用
- 选中的提交来自其他分支、远端跟踪分支或当前不可写的历史范围时，系统必须正确限制“修改当前历史”的动作
- 当前仓库存在未提交改动、进行中的变基 / 合并 / 摘取 / 回退流程时，危险动作必须阻止误操作或先引导用户处理当前流程
- 当前分支没有上游、所选提交不在当前分支可发布范围内，或用户并不位于当前分支时，“推送到这里”必须禁用并说明原因
- 当分支视图只加载了部分历史时，前往父提交 / 子提交不能把用户带到无上下文的空白状态；需要继续加载或清楚提示边界
- 创建分支或标签时若名称冲突，系统必须允许用户直接修正，而不是只返回底层报错
- 当动作导致冲突、被拒绝或部分完成时，系统不能只显示原始 Git 错误；必须说明当前仓库处于什么状态、接下来能做什么
- 当用户从分支列表打开分支菜单时，触发源分支行必须保持可辨认；不能出现“菜单开了，但不知道是给哪条分支开的”
- 当用户在历史视图中右键一条提交时，系统不能毫无反馈，也不能要求必须先切到分支视图才能完成基础提交动作
- 当提交标题、文件路径、标签装饰或比较摘要很长时，系统不能为了容纳文本而在同一阅读区域堆叠出多个横向滚动条；需要通过裁切、折叠、单一滚动层或等价方式保持界面整洁
- 当用户选中的是新增文件、未跟踪文件、空文件或二进制文件时，预览区必须给出正确预览：文本文件直接显示内容，空文件明确说明为空，二进制文件则给出可理解的占位说明

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 让分支视图中的可见提交成为一等交互对象，用户能够直接选中某条提交并对其发起动作
- **FR-002**: 系统 MUST 为提交提供稳定可发现的动作入口，例如右键菜单、悬浮动作或等价交互，不要求用户离开分支视图
- **FR-003**: 系统 MUST 按接近 PhpStorm / IDEA 的方式组织提交动作分组与顺序，使高频动作优先、危险动作靠后且有明显区分
- **FR-004**: 系统 MUST 使用完整中文标签、说明和状态提示来呈现提交动作，并在动作不可用时说明原因
- **FR-005**: 用户 MUST 能够从选中的提交直接复制版本号等基础标识信息
- **FR-006**: 用户 MUST 能够从选中的提交直接创建补丁
- **FR-007**: 用户 MUST 能够以选中的提交为基准，与当前本地上下文发起比较并查看差异
- **FR-008**: 用户 MUST 能够从选中的提交发起签出或等价的上下文切换动作，并在执行前理解对当前工作状态的影响
- **FR-009**: 用户 MUST 能够以选中的提交为起点创建新分支或新标签
- **FR-010**: 系统 MUST 支持在分支视图中从当前选中提交前往父提交或子提交，并保持导航结果可理解
- **FR-011**: 用户 MUST 能够从选中的提交对当前分支发起摘取或等价的“应用此提交”动作
- **FR-012**: 用户 MUST 能够从选中的提交直接发起回退该提交的动作
- **FR-013**: 用户 MUST 能够将当前分支重置到选中的提交；在真正执行前，系统 MUST 说明其对当前分支指针、工作区和本地提交的影响
- **FR-014**: 系统 MUST 支持“仅将当前分支发布到此提交”为目标的动作，但该动作只能面向当前分支的上游，并且必须在不可执行时解释原因
- **FR-015**: 系统 MUST 为符合条件的本地未发布提交提供历史整理动作，包括撤销最近提交、修改提交说明、修补、压缩、合并、丢弃以及从此处开始整理
- **FR-016**: 系统 MUST 根据提交归属、是否已发布、是否位于当前分支、是否存在进行中的 Git 流程等上下文，动态决定提交动作的显示、启用或禁用状态
- **FR-017**: 当某个提交动作需要更高风险确认时，系统 MUST 在执行前明确说明影响范围，而不是直接执行
- **FR-018**: 当摘取、回退、重置、整理历史等动作产生冲突、暂停或待继续状态时，系统 MUST 提供继续处理路径，而不是让用户只看到一次性报错
- **FR-019**: 每次提交级动作完成后，系统 MUST 立即刷新分支视图中的提交列表、当前分支位置、关联分支 / 标签和同步状态
- **FR-020**: 对于复制、比较、建分支、建标签等非破坏性动作，系统 MUST 尽量保留当前分支视图的滚动位置、选中提交和上下文
- **FR-021**: 系统 MUST 在用户从提交动作返回后，允许其继续沿着当前历史上下文操作，而不是强制跳回仓库首页或其他页面
- **FR-022**: 当某些动作因合并提交、多父提交、根提交或历史边界而存在歧义时，系统 MUST 明确提示用户如何继续，而不是自行做不透明猜测
- **FR-023**: 系统 MUST 为分支视图中的时间线、详情区、比较区和其他内嵌滚动容器提供低干扰、贴边、接近 PhpStorm 的滚动条视觉，不得让滚动条遮挡正文或成为主视觉噪音
- **FR-024**: 当分支视图中的内容发生横向溢出时，系统 MUST 保证同一阅读区域只存在一个清晰可理解的横向滚动机制，不得出现两个或更多叠压在内容上的横向滚动条
- **FR-025**: 系统 MUST 优先通过合理的文本裁切、换行、装饰折叠或容器层级调整来减少无意义横向滚动，而不是默认让长文本把滚动条推到内容中央
- **FR-026**: 当用户在相关文件列表或比较入口中选中新增文件或未跟踪文件时，系统 MUST 在右侧预览区直接显示该文件内容，而不是把它当作“没有变更”的空状态
- **FR-027**: 对新增文件或未跟踪文件的预览，系统 MUST 在文本文件场景下保持与普通 diff 预览一致的基本可读性，包括文件标题、路径、文件状态与正文展示；若无法做 diff，也要显示“整文件新增”的语义
- **FR-028**: 当新增文件不可直接以文本方式展示时，系统 MUST 提供清楚的替代反馈，例如“空文件”“二进制文件无法直接预览”或等价中文说明，而不是留下一块误导性的空白区域
- **FR-029**: 系统 MUST 让分支导航列表中的当前分支、选中分支与已打开上下文菜单的分支在视觉上可区分，同时保持接近 PhpStorm 的紧凑列表节奏
- **FR-030**: 系统 MUST 为分支列表行和文件夹行提供稳定可发现的上下文入口，至少支持右键打开，以及与列表密度匹配的紧凑行尾动作按钮
- **FR-031**: 系统 MUST 让分支动作菜单与提交动作菜单共享同一套弹层语言，包括相近的头部摘要、分组节奏、禁用原因表达与危险动作区分
- **FR-032**: 系统 MUST 允许用户在历史视图中对提交行使用右键打开基础提交菜单，并在该菜单中完成复制哈希、导出补丁或查看详情等高频非破坏性动作
- **FR-033**: 当用户从分支列表或历史列表打开上下文菜单时，系统 MUST 保持触发源行的可辨认状态，直到菜单关闭或上下文切换

### Key Entities *(include if feature involves data)*

- **Branch Timeline Surface**: 分支视图中的提交历史区域，承载提交列表、选中态、上下文菜单和提交导航
- **Commit Action Context**: 围绕某条提交展开的可执行动作集合，以及它们对应的说明、风险等级和启用条件
- **Commit Eligibility State**: 某条提交在当前仓库上下文下是否允许被签出、摘取、回退、重置、整理或发布的判定结果
- **Rewrite Range**: 当前分支中允许执行改说明、合并、压缩、丢弃、撤销提交等历史整理动作的一段本地提交范围
- **Publication Target**: “推送到这里”所对应的当前分支上游发布目标与可发布范围
- **Follow-up Flow State**: 某个动作执行后仓库进入的后续状态，例如冲突中、待继续、待确认完成或已恢复安全状态

## Assumptions

- 用户希望本次 spec 聚焦“分支视图里的提交级能力补齐”，目标交互参考其提供的 PhpStorm / IDEA 截图，而不是泛化成新的独立日志模块
- “推送到这里”仅面向当前分支及其已配置的上游关系，不扩展为多远端、多分支选择器
- 历史整理类动作主要面向当前分支上的本地未发布提交；对于已发布提交、远端提交或其他分支提交，默认以禁用或强提示为主
- 当某条提交存在多个父提交或多个可能的导航目标时，产品可以要求用户明确选择，而不是默认猜测其意图
- 分支视图应保持“看历史并顺手操作”的轻量定位；更重的全局日志分析仍可由独立历史入口承担

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户在分支视图中看到一条目标提交后，应能在 2 次交互内打开其动作菜单或等价入口
- **SC-002**: 对于复制版本号、创建补丁、基于提交建分支、与当前本地上下文比较这四类高频动作，用户应能在 10 秒内从看到目标提交走到发起操作
- **SC-003**: 对于当前分支上的本地未发布提交，至少 80% 的日常历史整理需求应可直接在分支视图中发起，而不依赖终端
- **SC-004**: 所有会移动分支指针、重写历史或发布到远端的动作，在执行前都必须向用户展示明确影响说明，避免“点了才知道发生了什么”
- **SC-005**: 当提交级动作成功完成后，分支视图应在用户可感知的短时间内反映新的提交顺序、当前分支位置和同步状态，避免用户基于旧状态继续操作
- **SC-006**: 在手工验收中，摘取、回退、重置、从此处开始整理等高风险动作若进入冲突或待继续状态，系统每次都能给出下一步入口，而不是只留下原始报错
- **SC-007**: 用户在查看历史并连续完成两个以上提交级动作时，应始终能够留在分支视图主线中继续操作，不需要频繁退回终端或独立历史页
- **SC-008**: 在手工验收中，分支视图的主要阅读区域不应再出现叠压正文、彼此重复或明显破坏视觉层次的双横向滚动条；滚动条应保持细窄、贴边且不抢走注意力
- **SC-009**: 在手工验收中，用户选中新建或未跟踪文件后，右侧预览区应立即显示文件内容或明确占位说明，不再错误出现“没有变更”的空状态
- **SC-010**: 在手工验收中，左侧分支列表里的当前分支、选中分支和菜单打开分支应能被快速辨认，且行尾上下文按钮不会破坏整体列表密度
- **SC-011**: 在手工验收中，用户在历史视图中右键任意可见提交时，应始终得到可执行的基础提交菜单，而不是无响应
