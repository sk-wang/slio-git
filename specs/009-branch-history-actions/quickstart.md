# Quickstart: 分支视图提交动作补齐

## 1. Build & Verification

实现完成后执行：

```bash
cargo check -p src-ui
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
```

## 2. Launch the UI

```bash
cargo run -p src-ui
```

启动后先确认：

- 中文界面、中文字体与状态提示都正常
- 能正常打开仓库并进入主工作区
- 从主工作区可进入分支视图

## 3. Branch Navigator Walkthrough

1. 打开一个包含本地分支、远程分支和至少一个分组文件夹节点的仓库。
2. 打开分支视图，先观察左侧分支列表。
3. 依次验证：
   - 当前分支有持续但不过分刺眼的识别提示
   - 手动选中另一条分支后，选中态与当前分支态能明显区分
   - 通过右键或行尾上下文按钮打开某条分支菜单后，触发源分支仍保持可辨认
   - 文件夹节点展开 / 折叠时，箭头、文本和整体行高都保持紧凑
   - 行尾上下文按钮可发现，但不会把列表节奏撑得很松散
4. 对任意本地分支和远程分支分别打开菜单，确认：
   - 分支菜单不是“点击没反应”
   - 菜单头部会明确显示目标分支
   - 菜单分组顺序与提交菜单接近，保持常用到危险动作的节奏
   - 禁用动作有中文原因，而不是静默消失

## 4. Branch Timeline Walkthrough

1. 在分支视图中选择一个本地分支。
2. 确认右侧不再只有摘要，而是能看到该分支对应的提交时间线。
3. 在时间线中切换几条提交，确认：
   - 选中态稳定
   - 图谱不乱位
   - 滚动位置不会频繁跳动
   - 当前提交的动作入口可发现
   - 右键和行尾按钮都能打开提交动作菜单
   - 没有出现叠在正文上的双横向滚动条，滚动条保持细窄贴边
4. 若当前场景包含新增文件或未跟踪文件，选中它后确认右侧预览区会直接显示文件内容，或在空文件 / 二进制文件场景下给出明确中文说明，而不是显示“没有变更”。

## 5. Non-Destructive Commit Actions Walkthrough

1. 对任意一条提交打开动作菜单。
2. 依次验证：
   - 复制版本号后有成功反馈
   - 创建补丁可以选择导出位置并生成 patch 文件
   - 基于提交创建分支后，分支列表和提交装饰会刷新
   - 基于提交创建标签后，标签装饰或标签列表会刷新
   - 菜单头部、分组标题、说明文字与分支菜单保持同一种 JetBrains 式弹层语言
3. 确认完成上述动作后，仍停留在分支视图上下文中。

## 6. Compare & Navigation Walkthrough

1. 选中一条非 HEAD 提交。
2. 验证可以：
   - 与当前本地上下文比较
   - 跳到父提交
   - 跳到子提交（若存在）
   - 临时 checkout 到该提交前先看到影响提示
3. 返回分支视图后，确认仍能继续围绕当前历史上下文操作。
4. 若比较或预览对象是新增文件，确认预览区按“整文件新增”语义展示，而不是空白。

## 7. Current-Branch Mutation Walkthrough

1. 在测试仓库准备若干适合演示的提交：
   - 可 cherry-pick 的提交
   - 可 revert 的提交
   - 当前分支可 reset 到的祖先提交
2. 在分支视图对这些提交分别尝试：
   - 摘取
   - 回退
   - 重置到这里
3. 确认每个高风险动作在真正执行前都会解释影响范围。
4. 对至少一个动作制造冲突，确认系统不会只给底层报错，而是提供继续处理入口。

## 8. Push-To-Here Walkthrough

1. 准备一个已配置 upstream 的当前分支，并确保本地有多条尚未发布提交。
2. 在分支视图中选择其中一条祖先提交。
3. 触发“推送到这里”，确认：
   - 只针对当前分支上游生效
   - 不可执行时会明确说明原因
   - 成功后同步状态会刷新
4. 再准备一个无 upstream 或目标不在当前分支范围内的场景，确认动作会被禁用或阻止。

## 9. Rewrite Walkthrough

1. 在当前分支上准备 3~5 条本地未发布提交。
2. 在分支视图中验证：
   - 修改提交说明
   - fixup / squash / drop
   - 从这里开始整理
   - 撤销最近提交
3. 对 rewrite 过程制造冲突或暂停，确认可进入继续 / 跳过 / 中止路径。
4. 对已发布提交、merge commit、远端提交再次检查，确认 rewrite 动作会被禁用或要求更强确认。

## 10. History View Basic Menu Walkthrough

1. 打开独立历史视图，并确保当前至少能看到数条提交。
2. 右键任意一条可见提交，确认：
   - 会立刻弹出基础提交菜单，而不是无响应
   - 触发源提交行保持可辨认
   - 菜单至少包含“复制哈希”“导出 Patch”“查看详情”
3. 依次执行：
   - 复制哈希，确认剪贴板与成功反馈正常
   - 导出 Patch，确认可选择导出位置并生成文件
   - 查看详情，确认详情刷新但仍停留在历史视图上下文
4. 关闭菜单后再次切换提交，确认旧的菜单源高亮会清掉，不会残留错位状态。

## 11. Edge-Case Walkthrough

至少手工验证以下场景：

- 根提交：不能错误显示需要父提交的动作
- merge commit：需要区分是否允许回退、是否需要额外说明
- 当前仓库有未提交改动：危险动作要明确阻止或给出前置说明
- 当前处于 rebase / merge / cherry-pick / revert 中：不能再随意发起新的同类危险动作
- 历史列表只加载了部分提交：导航到边界时要给出清楚提示

## 12. Acceptance Mapping

| Success Criteria | Verification Focus |
|------------------|--------------------|
| SC-001 | 看到目标提交后，是否能在 2 次交互内打开动作入口 |
| SC-002 | 复制版本号、创建补丁、基于提交建分支、比较是否都能在 10 秒内发起 |
| SC-003 | 本地未发布提交的大多数整理需求是否可直接在分支视图发起 |
| SC-004 | 所有危险动作是否都在执行前明确解释影响 |
| SC-005 | 动作完成后提交顺序、分支位置和同步状态是否及时刷新 |
| SC-006 | 冲突 / 待继续场景下是否总能看到下一步入口 |
| SC-007 | 用户是否能连续完成多个提交级动作而不退出分支视图主线 |
| SC-008 | 分支视图主要阅读区域是否只保留单一、细窄且不压正文的滚动机制 |
| SC-009 | 新增 / 未跟踪文件是否始终能预览正文或得到明确占位说明 |
| SC-010 | 左侧分支列表里的当前 / 选中 / 菜单打开状态是否都能被快速辨认 |
| SC-011 | 历史视图右键任意提交时，是否始终能得到基础提交菜单而不是无响应 |
