## 用户场景与测试 *(必填)*

### 用户故事 1 - 仓库管理 (优先级: P1)

用户可以打开或初始化 git 仓库，应用程序自动检测仓库状态，包括当前分支、跟踪的文件和远程配置。

**优先级原因**: 仓库管理是一切的基础——没有它，任何 git 操作都无法进行。这复刻了 IntelliJ 的自动仓库检测功能。

**独立测试**: 可以通过打开一个已存在的 git 仓库并验证所有状态信息是否正确检测和显示来测试。

**验收场景**:

1. **假设** 目录中存在一个已存在的 git 仓库，**当** 用户打开该目录，**那么** 应用程序在 2 秒内显示当前分支、文件状态和远程配置
2. **假设** 目录中没有 git 仓库，**当** 用户选择"初始化仓库"，**那么** 创建新的 git 仓库，用户可以执行提交操作
3. **假设** 存在嵌套 git 仓库（子模块或 worktree），**当** 用户打开目录，**那么** 应用程序正确识别仓库根目录并独立管理

---

### 用户故事 2 - 文件变更与暂存 (优先级: P1)

用户可以查看所有文件变更（已修改、已暂存、未跟踪），并可以暂存/取消暂存单个文件或代码块，完全匹配 IntelliJ 的行为。

**优先级原因**: 变更视图是主要的日常使用界面。用户必须能够立即看到文件状态，并以与 IntelliJ 完全相同的交互方式暂存变更。

**独立测试**: 可以通过修改仓库中的文件并验证变更检测和暂存行为是否与 IntelliJ 一致来测试。

**验收场景**:

1. **假设** 存在未暂存的变更文件，**当** 用户查看变更面板，**那么** 所有修改的文件都列出，并标有正确的状态指示（已修改/已暂存/未跟踪）
2. **假设** 一个已修改的文件，**当** 用户暂存它（单个文件或代码块），**那么** 文件移动到暂存区并显示正确的 diff
3. **假设** 一个已暂存的文件，**当** 用户取消暂存，**那么** 文件返回到未暂存区
4. **假设** 一个冲突文件（存在 MERGE_HEAD），**当** 用户查看变更，**那么** 冲突文件被清晰标记，diff 显示所有三个版本（ ours/theirs/base）

---

### 用户故事 3 - 提交操作 (优先级: P1)

用户可以创建提交（带消息编辑）、修改前一次提交，并以与 IntelliJ 完全相同的交互模式查看提交历史。

**优先级原因**: 提交是核心 git 操作——必须匹配 IntelliJ 提交对话框的行为，包括 diff 预览、提交消息处理和修改流程。

**独立测试**: 可以通过使用各种消息格式创建提交并验证生成的 git 对象是否符合预期状态来测试。

**验收场景**:

1. **假设** 有已暂存的文件且提交消息为空，**当** 用户打开提交对话框，**那么** 显示已暂存变更的 diff，提交按钮被禁用直到输入消息
2. **假设** 有已暂存的文件和有效的提交消息，**当** 用户点击提交，**那么** 创建具有正确哈希、作者和消息的提交
3. **假设** 上一次提交且没有变更被暂存，**当** 用户选择"修改"，**那么** 前一次提交的消息被预填充并可以修改
4. **假设** 提交历史视图，**当** 用户滚动，**那么** 提交加载作者、日期、消息和变更文件数量，与 git log 输出匹配

---

### 用户故事 4 - 分支操作 (优先级: P1)

用户可以创建、切换、合并、删除和重命名分支，使用与 IntelliJ 分支对话框完全相同的弹出窗口和工作流程。

**优先级原因**: 分支管理是日常工作的关键——IntelliJ 的分支弹出窗口经过高度优化，必须精确复刻。

**独立测试**: 可以通过执行分支操作并验证仓库状态是否符合预期的分支配置来测试。

**验收场景**:

1. **假设** 仓库有多个分支，**当** 用户打开分支弹出窗口，**那么** 所有本地和远程分支都列出，当前分支被标记
2. **假设** 一个本地分支，**当** 用户检出它，**那么** HEAD 指针移动，工作树立即更新
3. **假设** 两个分支有可合并的内容，**当** 用户选择合并，**那么** 创建具有正确父提交的新合并提交
4. **假设** 一个已完全合并的分支，**当** 用户删除它，**那么** 分支引用从 .git/refs/heads/ 中移除

---

### 用户故事 5 - 远程操作 (优先级: P2)

用户可以向/从远程仓库执行 fetch、pull、push 操作，认证处理与 IntelliJ 完全相同。

**优先级原因**: 协作需要远程操作——IntelliJ 对 SSH 密钥、凭据助手和进度报告的处理必须匹配。

**独立测试**: 可以通过针对测试远程执行远程操作并验证正确的数据传输和分支跟踪来测试。

**验收场景**:

1. **假设** 远程仓库有新提交，**当** 用户 fetch，**那么** 远程引用更新，新提交在典型仓库大小下 5 秒内出现在日志中
2. **假设** 工作树干净且远程有新提交，**当** 用户 pull，**那么** 本地分支快速前进来匹配远程
3. **假设** 本地提交领先于远程，**当** 用户 push，**那么** 提交被传送到远程，本地分支更新
4. **假设** 认证失败，**当** 用户尝试远程操作，**那么** 凭据提示出现，匹配系统原生对话框风格

---

### 用户故事 6 - 储藏管理 (优先级: P2)

用户可以储藏变更、列出储藏、应用或删除储藏，与 IntelliJ 储藏面板的行为完全相同。

**优先级原因**: 储藏是在切换上下文时必不可少的——IntelliJ 的储藏管理（包括选择性储藏）被广泛使用。

**独立测试**: 可以通过储藏变更、验证储藏列表和应用/删除储藏来测试。

**验收场景**:

1. **假设** 有未暂存的变更，**当** 用户创建储藏，**那么** 工作树变干净，储藏出现在储藏列表中并带有描述
2. **假设** 存在现有储藏，**当** 用户查看储藏列表，**那么** 所有储藏都显示消息和时间戳
3. **假设** 一个储藏，**当** 用户应用它，**那么** 变更重新出现在工作树中（应用到当前分支状态）
4. **假设** 一个储藏，**当** 用户删除它，**那么** 储藏从 .git logs 中移除

---

### 用户故事 7 - Diff 与历史查看器 (优先级: P2)

用户可以查看文件和提交 diff、浏览历史和搜索提交，与 IntelliJ 的 diff 查看器行为一致。

**优先级原因**: 代码审查和历史导航是日常活动——IntelliJ 的统一和分屏 diff 视图必须匹配。

**独立测试**: 可以通过查看提交和文件的 diff、验证视觉呈现是否符合预期的 diff 格式来测试。

**验收场景**:

1. **假设** 一个提交，**当** 用户查看其 diff，**那么** 所有变更的文件列出，增加/删除计数
2. **假设** 一个有变更的文件，**当** 用户查看 diff，**那么** 分屏和统一 diff 视图可用，高亮与 git diff 一致
3. **假设** 历史视图，**当** 用户按消息搜索提交，**那么** 显示匹配的提交并正确过滤
4. **假设** 两个提交，**当** 用户比较它们，**那么** 显示提交之间的 diff 和正确的文件变更

---

### 用户故事 8 - 变基操作 (优先级: P3)

用户可以执行交互式变基，使用与 IntelliJ 相同的基于编辑器的工作流程。

**优先级原因**: 交互式变基是用于提交清理的高级功能——IntelliJ 委托给编辑器，必须复刻。

**独立测试**: 可以通过执行交互式变基操作并验证最终的提交历史来测试。

**验收场景**:

1. **假设** 一系列提交，**当** 用户开始交互式变基，**那么** 变基待办列表以正确格式在编辑器中呈现
2. **假设** 修改变基待办（重排序、压缩、改写、删除），**当** 用户保存并关闭编辑器，**那么** 变基执行，历史反映新的提交顺序
3. **假设** 变基进行中且有冲突，**当** 用户解决冲突，**那么** 变基继续完成，最终状态正确

### 用户故事 9 - 标签操作 (优先级: P3)

用户可以创建、删除和查看标签，与 IntelliJ 的工作流程相同。

**优先级原因**: 标签用于发布——简单操作但必须匹配 IntelliJ 的行为。

**独立测试**: 可以通过创建带注释和轻量级标签并验证它们出现在 git tag 输出中来测试。

**验收场景**:

2. **假设** 一个现有标签，**当** 用户查看标签详情，**那么** 带注释的标签显示标签消息和标签创建者

- **FR-003**: 系统必须能够暂存单个文件、代码块和所有文件，行为与 IntelliJ 完全相同
- **FR-004**: 系统必须创建具有作者、消息、时间戳的提交，与 IntelliJ 的 git 插件行为一致
- **FR-005**: 系统必须支持提交修改，具有相同的消息预填充行为
- **FR-006**: 系统必须显示提交历史，包括作者、日期、消息、变更文件数量，与 `git log` 输出一致
- **FR-007**: 系统必须列出、创建、切换、重命名、删除和合并分支，与 IntelliJ 的交互流程相同
- **FR-008**: 系统必须执行 fetch、pull、push 操作，进度报告与 IntelliJ 完全一致
- **FR-009**: 系统必须处理 SSH 密钥和凭据助手认证，提示行为与 IntelliJ 相同
- **FR-010**: 系统必须创建、列出、应用和删除储藏，与 IntelliJ 的工作流程相同
- **FR-011**: 系统必须在统一和分屏视图中显示 diff，语法高亮与 IntelliJ 的 diff 查看器一致
- **FR-012**: 系统必须通过编辑器支持交互式变基，命令格式与 IntelliJ 相同
- **FR-013**: 系统必须创建、删除和列出标签（带注释和轻量级）
- **FR-014**: 系统必须检测并显示合并冲突，具有三向 diff（ours/theirs/base）
- **FR-015**: 系统必须在 .git 目录更改时通过文件监视实时更新仓库状态
- **FR-016**: 系统必须使用与 IntelliJ 相同的方式处理 git worktree 的检测和管理
- **FR-017**: 用户界面必须使用中文显示，与 IntelliJ 的本地化行为一致
- **FR-018**: slio-git 必须复刻 IntelliJ 的核心 git 操作算法和 UI 布局，包括主要窗口结构、面板布局和操作流程
- **FR-019**: 系统必须使用 Pure Iced 架构，不使用 Tauri、WebView 或任何 Electron 类似的运行时

### 关键实体 *(涉及数据时必填)*

- **仓库 (Repository)**: git 项目的根目录，包含 .git 目录，由 slio-git 跟踪状态管理
- **分支 (Branch)**: 指向提交的命名引用，支持本地和远程跟踪分支
- **提交 (Commit)**: 仓库状态的快照，包含作者、消息、时间戳、父提交和树
- **远程 (Remote)**: 命名的远程（origin 等），带有 URL 和 fetch/push refspecs
- **储藏 (Stash)**: 保存在 .git/logs/refs/stash 中的保存工作树状态
- **标签 (Tag)**: 指向提交的命名引用，可以是带注释的（带消息）或轻量级的
- **索引条目 (IndexEntry)**: 暂存区中的单个文件条目，包含路径、哈希和权限
- **变更 (Change)**: 文件修改，包含状态（已修改/已暂存/未跟踪/冲突）和 diff 内容

## 成功标准 *(必填)*

### 可衡量指标

- **SC-001**: 包含 10,000 个文件的仓库在标准硬件上打开并在 2 秒内显示状态
- **SC-002**: 100% 的 git 操作（提交、分支、合并、储藏）完成结果与相同输入的 git CLI 完全一致
- **SC-003**: Diff 查看器在 1 秒内渲染 10,000+ 行的文件，UI 不会冻结
- **SC-004**: 对于有 100+ 分支的仓库，分支检出在 500ms 内完成
- **SC-005**: 通过文件监视检测到的文件系统更改在 200ms 内更新 UI
- **SC-006**: Push/fetch 操作以与 IntelliJ 相同的粒度显示实时进度（传输的字节/对象）
- **SC-007**: 95% 的用户交互在 100ms 内产生 UI 响应（不包括实际的 git 操作）
- **SC-008**: 应用程序启动到可用状态（仓库已加载）在 300ms 内 (Pure Rust，无运行时开销)
- **SC-009**: 在正常操作期间处理 50,000+ 提交的仓库时内存使用低于 80MB (无 WebView/JS 运行时)
