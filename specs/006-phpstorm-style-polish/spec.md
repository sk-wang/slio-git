# Feature Specification: PhpStorm 风格的轻量化样式收敛

**Feature Branch**: `006-phpstorm-style-polish`  
**Created**: 2026-03-23  
**Status**: Draft  
**Input**: User description: "还是太重了现在这个交互，以这个图为标准，优化UI的样式"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 第一屏像 PhpStorm 一样轻 (Priority: P1)

作为用户，我希望打开仓库后的主工作区像参考图那样，顶部和侧边都更薄、更克制，主要视觉面积留给改动树和差异内容，而不是被大量卡片、圆角块、说明文字和大间距吃掉。

**Why this priority**: 用户当前最强烈的不满是“太重”，问题首先是样式密度和视觉重量，而不是功能缺失。

**Independent Test**: 打开任意仓库后，只看第一屏即可判断界面是否明显更贴近参考图：主内容连续、边框更薄、卡片感减弱、上下留白减少，改动树和 diff 成为视觉中心。

**Acceptance Scenarios**:

1. **Given** 用户已打开仓库，**When** 主工作区完成渲染，**Then** 顶部上下文、工具入口和主体区域形成连续的暗色工作台，而不是多个厚重分离卡片堆叠
2. **Given** 用户查看改动工作区，**When** 比较视觉比例，**Then** 改动列表与差异内容占据大部分可见面积，顶部与状态区保持细窄
3. **Given** 用户浏览界面，**When** 观察圆角、边框、阴影和留白，**Then** 样式以细边界、轻分割和紧凑间距为主，不再出现明显“组件盒子套盒子”的厚重感

---

### User Story 2 - 分支和动作弹层像 JetBrains 原生面板 (Priority: P1)

作为用户，我希望当前分支入口弹出的面板更像参考图中的 JetBrains 分支面板：一个紧凑、连续、以列表为核心的弹层，而不是信息块很多、段落很多、操作过于分散的管理页。

**Why this priority**: 参考图里最明确的交互锚点就是分支弹层；如果这里仍然偏“表单页/卡片页”，整体气质就会继续偏重。

**Independent Test**: 点击当前分支入口后，用户可以在一个轻量列表式弹层中搜索、查看最近/本地/远程分支，并执行高频 Git 操作，且弹层不会像完整页面一样占据过多解释性内容。

**Acceptance Scenarios**:

1. **Given** 用户点击当前分支入口，**When** 面板展开，**Then** 面板以搜索框、紧凑动作项和列表分组为主，而不是大块说明、统计卡片和重复摘要
2. **Given** 用户查看分支列表，**When** 面板显示最近、本地和远程分组，**Then** 每一项主要展示分支名、必要跟踪信息和选中态，不堆叠冗余文案
3. **Given** 用户需要执行刷新、提交、拉取、推送或创建分支，**When** 查看弹层，**Then** 高频动作以更接近 JetBrains 菜单/列表项的密度呈现，并与分支列表保持统一视觉节奏

---

### User Story 3 - 列表、标签和状态元素更克制 (Priority: P2)

作为用户，我希望文件列表、标签、计数、状态提示和底部区域都更像参考图里的 IDE 元素：薄、紧、平，不要出现太多醒目的徽标块、厚按钮和高对比装饰。

**Why this priority**: 即便主布局已经收敛，如果文件项、徽标和状态区仍然过厚，整体仍会显得“重”。

**Independent Test**: 在改动列表、diff 顶栏、状态栏和底部区域回归时，用户能感知到元素更贴近 IDE 控件风格：更小的高度、更少的填充、更弱的装饰、更连续的背景。

**Acceptance Scenarios**:

1. **Given** 用户查看改动列表，**When** 对比每个文件项、分组标题和操作按钮，**Then** 行高和内边距明显收敛，列表密度更接近参考图
2. **Given** 用户查看状态标签、计数或提示，**When** 这些元素显示在界面上，**Then** 它们优先表现为轻量文本/小型标签，而不是高存在感的大徽章
3. **Given** 用户查看底部状态区和次要区域，**When** 界面处于稳定状态，**Then** 这些区域提供必要上下文但不抢占主视觉焦点

---

### User Story 4 - 更轻的样式下仍保留清晰可读性与可达性 (Priority: P2)

作为用户，我希望样式虽然更轻、更像参考图，但仍然能清晰区分当前选中项、当前分支、可点击动作、冲突/错误状态和主要操作入口，不会因为追求极简而变得难看懂或难点击。

**Why this priority**: 用户要的是“轻”，不是“弱化到看不清”；参考图本身也是通过选中态、分割线和层级关系来保持清晰度。

**Independent Test**: 在选中文件、切换分支、出现冲突或错误、查看空状态等场景下，用户仍能一眼识别焦点和关键动作，且所有核心 Git 能力保持可达。

**Acceptance Scenarios**:

1. **Given** 用户选中某个改动文件，**When** 观察左侧列表与右侧差异区，**Then** 选中态依然明确，但不依赖厚重边框或巨大高亮块
2. **Given** 用户遇到冲突、失败或需要注意的状态，**When** 系统提示这些状态，**Then** 界面仍会清晰强调，但只强调真正重要的内容
3. **Given** 用户从更轻的主工作区继续执行提交、分支、远程、标签、储藏或 rebase，**When** 用户查找入口，**Then** 能力路径仍然清晰可发现，不因样式收敛而被埋没

### Edge Cases

- 当仓库名、分支名或标签很长时，如何在保持紧凑的同时不让顶部入口和标签区变形
- 当没有改动、没有仓库或没有可执行动作时，如何保持空状态轻量而不是重新出现大段说明和大面积留白
- 当分支很多时，弹层如何保持 JetBrains 风格的轻量列表感，而不是退化为又长又重的管理页
- 当冲突、远程失败或 rebase 进行中时，如何在必要强调与整体轻量之间取得平衡
- 当窗口变窄时，如何优先保住改动树、diff 和当前上下文，而不是让次要样式元素先占空间

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 以用户提供的 PhpStorm Git/Changes 截图作为本次样式收敛的视觉基线，优先对齐其“轻量、连续、紧凑”的整体观感
- **FR-002**: 系统 MUST 将仓库工作区的持久顶部区域收敛为更薄的一层或两层连续工作条，不再呈现厚重的多层卡片化结构
- **FR-003**: 系统 MUST 让主工作区主体呈现为连续的内容平面，减少明显独立卡片、厚阴影和大圆角容器的数量
- **FR-004**: 系统 MUST 明显缩减界面主要区域的默认内边距、区块间距和控件高度，使整体信息密度更接近 JetBrains IDE
- **FR-005**: 系统 MUST 让改动列表与 diff 区域在默认窗口尺寸下成为绝对视觉主角，其他区域只承担辅助作用
- **FR-006**: 系统 MUST 将当前仓库与当前分支入口表现为更接近 JetBrains 的紧凑上下文控件，而不是大面积说明卡
- **FR-007**: 系统 MUST 将分支与动作弹层重构为以搜索、动作项和分组列表为核心的轻量面板，而不是完整管理页面式布局
- **FR-008**: 系统 MUST 在分支面板中优先显示当前分支、高频动作、最近分支、本地分支和远程分支，并以列表节奏统一呈现
- **FR-009**: 系统 MUST 控制分支面板中的说明文案、摘要块和重复状态表达，避免它们抢占列表可读面积
- **FR-010**: 系统 MUST 将状态标签、计数和反馈元素收敛为更小、更平、更低干扰的样式，只有关键风险状态可以更强强调
- **FR-011**: 系统 MUST 让左侧文件列表项、分组标题、操作按钮和底部状态区在行高、边框、留白和背景处理上更接近 IDE 控件风格
- **FR-012**: 系统 MUST 保持当前选中态、当前分支态、错误/冲突态和可点击动作的清晰可辨，不得因为视觉收敛而损失可读性
- **FR-013**: 系统 MUST 在无仓库、无改动和空列表场景下继续保持轻量风格，不得通过大尺寸空状态卡片重新引入“重”的观感
- **FR-014**: 系统 MUST 保留所有现有核心 Git 能力的可达性，包括提交、刷新、暂存、取消暂存、推送、拉取、分支、储藏、历史、标签、冲突处理与 rebase
- **FR-015**: 系统 MUST 让样式收敛同时覆盖主工作区、分支弹层、改动列表、diff 区、状态栏和底部区域，避免只局部变轻导致整体风格割裂
- **FR-016**: 系统 MUST 保持完整中文界面与中文可读性，不得为了贴近参考图而牺牲现有中文本地化体验

### Key Entities *(include if feature involves data)*

- **Visual Baseline**: 用户提供的 PhpStorm Git/Changes 截图，作为本次样式轻量化的主要参考标准
- **Compact Workspace Chrome**: 仓库打开后顶部、侧边和底部的持久 UI 外壳，要求更薄、更连续、更少卡片感
- **Branch Action Popup**: 从当前分支入口展开的轻量面板，用于承载搜索、分支列表和高频 Git 动作
- **Changes Pane**: 左侧改动树/文件列表区域，要求更高信息密度和更弱装饰感
- **Diff Pane**: 右侧差异预览区域，要求成为视觉重心并减少被外围样式抢占空间
- **Lightweight Status Surface**: 标签、计数、反馈与状态条组成的轻量状态层，只在必要时增强强调

## Assumptions

- 本次需求重点是“样式和视觉重量继续收敛”，不是重做 Git 交互算法
- 用户提供的截图主要用于约束样式密度、层次感和界面气质，而不是要求逐像素复刻 PhpStorm 全部 IDE 框架
- 现有功能入口可以继续保留，但它们的外观、层级和占位方式需要进一步向 JetBrains 风格靠拢
- 允许继续使用当前的主工作区结构，只要样式最终更轻、更连续、更像参考图
- 产品名称依然可以存在于窗口标题或系统层级，但不应重新回到仓库工作区主视觉中

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 打开仓库后的默认窗口尺寸下，主工作区顶部持久样式层不超过两条明显水平带，且总视觉高度明显低于当前版本
- **SC-002**: 在默认窗口尺寸下，改动列表与 diff 区合计占据主工作区可见面积的绝大部分，用户第一眼感知焦点仍然是内容而不是容器样式
- **SC-003**: 用户在 3 秒内可识别当前仓库、当前分支和当前改动区域，且不需要在多个厚重样式块之间来回确认
- **SC-004**: 分支与动作面板展开后，用户可在 5 秒内定位搜索入口、当前分支和至少一个高频动作，且面板整体更接近 JetBrains 列表式弹层
- **SC-005**: 在稳定工作状态下，主工作区中高存在感徽章、厚卡片和大面积说明块的数量较当前版本明显减少
- **SC-006**: 对打开仓库、切换分支、选择文件、刷新、暂存和提交等高频流程回归时，界面保持更轻的样式同时不降低任务完成率
- **SC-007**: 至少一轮人工走查可以明确指出样式收敛成果，包括“更薄的顶部区域”“更轻的分支弹层”“更紧凑的列表项”“更弱的卡片感”四类变化

## Implementation Notes

- 顶部仓库工作区已收敛为两层紧凑 chrome：上层承载仓库 / 分支上下文与高频动作，下层承载导航与次级 Git 能力入口
- `src-ui/src/theme.rs`、`src-ui/src/widgets/button.rs`、`src-ui/src/widgets/text_input.rs` 与 `src-ui/src/widgets/statusbar.rs` 已统一下调圆角、padding、badge 强度与 stable-state 状态面存在感
- `src-ui/src/widgets/changelist.rs` 与 diff widgets 已把文件行、分组标题、hunk 容器和 diff 顶栏压缩到更接近 JetBrains Changes 面板的密度
- `src-ui/src/views/branch_popup.rs` 已改成搜索驱动的列表 popup，并把最近 / 本地 / 远程分支与高频动作放到同一连续节奏中
- `src-ui/src/state.rs` 通过 `WorkspaceContextStrip`、`CompactChromeProfile` 和 `LightweightStatusSurface` 固化了更轻但仍清晰的状态模型
- 最终验收仍需按 `quickstart.md` 启动 UI 做截图基线比对；自动化门禁已覆盖 `cargo test` 与 `cargo clippy --workspace --all-targets -- -D warnings`
