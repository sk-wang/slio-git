# Feature Specification: IDEA 风格的极简 Git 工作台

**Feature Branch**: `005-idea-minimal-shell`  
**Created**: 2026-03-23  
**Status**: Draft  
**Input**: User description: "交互参考 IDEA 代码在 ~/git/intellij-community/ ，尽量精简交互和UI元素，现在太冗杂了，像顶部的slio-git这种完全没必要显示"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 更快进入改动处理主线 (Priority: P1)

作为用户，我希望打开仓库后第一眼只看到和当前 Git 工作直接相关的内容：仓库、分支、改动列表和差异，而不是大量标题、说明、芯片和重复入口，这样我能马上开始查看和处理改动。

**Why this priority**: 当前问题首先不是缺功能，而是主工作区被过多装饰性元素稀释，影响核心任务起步速度。

**Independent Test**: 打开任意仓库后，用户无需阅读多余说明即可在一个主工作区内识别当前仓库、当前分支、改动列表和差异区，并开始暂存、提交或切换分支。

**Acceptance Scenarios**:

1. **Given** 用户已打开仓库，**When** 主工作区渲染完成，**Then** 顶部只保留当前上下文和最关键操作，不再长期显示与当前任务无关的产品标题或宣传文案
2. **Given** 用户进入改动工作区，**When** 用户查看页面，**Then** 主体面积优先留给改动树和差异预览，而不是被多层工具条、芯片或说明块占用
3. **Given** 用户需要处理单个文件改动，**When** 用户选择文件，**Then** 选中态、差异区和可执行动作保持就近呈现，不需要在多个重复区域之间来回确认上下文

---

### User Story 2 - 像 IDEA 一样通过上下文切换器完成次要操作 (Priority: P1)

作为用户，我希望分支切换、常用 Git 动作和较少使用的操作都通过一个更克制的上下文入口集中呈现，类似 IDEA 的分支与动作弹层，而不是把所有动作平铺在主界面上，这样我能保持专注，同时仍然快速找到能力入口。

**Why this priority**: 用户明确要求参考 IDEA，并强调“尽量精简交互和 UI 元素”；这意味着次要动作应被收纳而不是持续占位。

**Independent Test**: 在打开仓库后，用户可以通过一个清晰的仓库/分支上下文入口进入分支与动作弹层，并完成刷新、提交、推送、拉取、创建分支等任务。

**Acceptance Scenarios**:

1. **Given** 用户正在查看当前仓库，**When** 用户点击当前分支或上下文入口，**Then** 系统展示一个可聚焦的分支与动作面板，而不是把所有相关操作永久平铺在工作区顶部
2. **Given** 分支数量较多，**When** 用户打开分支与动作面板，**Then** 系统提供快速搜索或筛选能力，帮助用户定位目标分支或操作
3. **Given** 用户需要执行次要操作，**When** 用户查看该面板，**Then** 常用动作、最近分支和本地/远程分组以清晰层级呈现，不干扰主工作区阅读

---

### User Story 3 - 减少重复反馈和视觉噪音 (Priority: P2)

作为用户，我希望界面不要在多个位置重复显示同一份信息，例如当前 section、下一步建议、仓库路径、状态提示等，而是用更节制的反馈方式只在必要时出现，这样我不会被反复打断。

**Why this priority**: 当前冗余感不仅来自按钮数量，也来自同一语义在标题、芯片、banner、侧边栏和正文中重复出现。

**Independent Test**: 对打开仓库、刷新、暂存、取消暂存、切换分支等常见操作进行回归时，界面只在需要时显示短时反馈，平稳状态下不出现多处重复上下文说明。

**Acceptance Scenarios**:

1. **Given** 用户处于稳定工作状态，**When** 没有新的错误或结果需要强调，**Then** 页面不持续展示重复性的引导性横幅或冗长说明
2. **Given** 用户完成一次操作，**When** 系统需要反馈结果，**Then** 反馈以简洁、短时、非侵入的形式呈现，并在完成后让主工作区恢复简洁状态
3. **Given** 页面已经在某处明确表达当前仓库、分支或选中内容，**When** 用户继续操作，**Then** 系统不会在其他持久区域重复堆叠相同信息

---

### User Story 4 - 在精简后仍保留全部核心 Git 可达性 (Priority: P2)

作为用户，我希望界面更简洁，但不希望因为“简化”而失去提交、储藏、历史、标签、远程、冲突和 rebase 等现有能力，这样我既能获得更克制的 UI，也不会丢失工具价值。

**Why this priority**: 简化不是砍功能，而是重新组织入口；如果能力变得难找或不可达，重构就会失败。

**Independent Test**: 从更精简的主工作区出发，用户仍然可以在少量步骤内进入所有现有 Git 能力，并完成关键流程。

**Acceptance Scenarios**:

1. **Given** 用户需要执行提交、储藏、历史、标签、远程、冲突处理或 rebase，**When** 用户从主工作区发起操作，**Then** 系统在不增加常驻噪音的前提下提供清晰入口
2. **Given** 某项能力被从主界面移入弹层、菜单或二级入口，**When** 用户首次使用新版界面，**Then** 仍能在合理时间内发现该能力并理解其所在位置
3. **Given** 用户关闭某个辅助面板或完成一次操作，**When** 用户返回主工作区，**Then** 界面立即回到简洁、聚焦的默认状态

### Edge Cases

- 当仓库处于 detached HEAD、无上游或新仓库状态时，顶部上下文如何保持简洁，同时不隐藏关键状态
- 当分支名、仓库名或路径很长时，如何保证上下文入口仍可读，但不会撑坏顶部布局
- 当仓库分支很多、远程很多或最近操作很多时，弹层如何继续保持轻量，不演变成新的信息噪音源
- 当当前没有仓库、没有变更或没有可执行动作时，如何保持空状态简洁，而不是用大段说明占满页面
- 当冲突、rebase 或远程失败等高优先级状态出现时，如何在必要提示与常态简洁之间取得平衡

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 将“当前仓库 + 当前分支”作为打开仓库后的主上下文入口，并把它放在用户进入主工作区后最容易注意到的位置
- **FR-002**: 系统 MUST 在仓库已打开的主工作区中移除长期常驻的产品名、标语或与当前 Git 任务无直接关系的装饰性标题
- **FR-003**: 系统 MUST 将主工作区的持久可见元素收敛到核心任务所需范围，避免在同一层级同时长期显示过多按钮、芯片、说明和状态块
- **FR-004**: 系统 MUST 为分支切换与次要 Git 操作提供一个集中式上下文面板或等价入口，使用户不必依赖满屏常驻按钮
- **FR-005**: 系统 MUST 在分支与动作入口中优先呈现当前分支、最近使用项和最常见操作，并将较少使用能力放在次级层级
- **FR-006**: 当分支或动作数量较多时，系统 MUST 提供搜索、筛选或快速定位能力，避免用户在长列表中机械滚动
- **FR-007**: 系统 MUST 保证改动列表与差异预览继续作为主工作区的视觉重点，不被新的上下文或导航元素挤压到次要位置
- **FR-008**: 系统 MUST 减少重复语义在多个持久区域中的重复出现；同一时间内，仓库、分支、section、下一步建议等上下文信息不应在多个固定区域重复堆叠
- **FR-009**: 系统 MUST 将成功、失败、警告和处理中反馈收敛为更短、更聚焦的表达方式，并在状态恢复稳定后退出持久占位
- **FR-010**: 系统 MUST 保留当前已有 Git 能力的可达性，包括但不限于提交、刷新、暂存、取消暂存、推送、拉取、分支、储藏、历史、标签、冲突处理与 rebase
- **FR-011**: 系统 MUST 允许将非主线能力迁移到弹层、菜单、二级页或上下文入口中，只要这些能力仍可被清晰发现并完成
- **FR-012**: 系统 MUST 在无仓库、无变更、无可执行动作和异常状态下继续保持简洁表达，不得因为空状态而重新引入大段冗余说明
- **FR-013**: 系统 MUST 让用户在仓库打开后的首屏中，仅通过一个主上下文区域识别当前仓库与分支，不必在顶部、侧栏、正文和状态栏之间来回对照
- **FR-014**: 系统 MUST 以 IDEA 的 Git/Changes 交互节奏作为参考：主界面聚焦改动处理，次要能力通过上下文切换器或弹层渐进展开
- **FR-015**: 系统 MUST 保持精简后的交互在常见窗口尺寸下仍然稳定，且不能通过隐藏关键能力来换取表面上的“清爽”

### Key Entities *(include if feature involves data)*

- **Workspace Context Switcher**: 主工作区中的唯一高优先级上下文入口，用来表达当前仓库、当前分支以及进入分支/动作面板的入口
- **Primary Changes Workspace**: 用户处理改动的核心区域，承载改动列表、差异预览和与当前选中内容最相关的操作
- **Branch & Actions Panel**: 用于承载分支切换、搜索、最近项和常见 Git 动作的集中式上下文面板
- **Secondary Action Entry**: 对历史、标签、储藏、远程、rebase 等非主线能力的精简入口，默认不长期占据主工作区视觉中心
- **Minimal Feedback Layer**: 只在必要时出现的反馈层，用于表达成功、失败、警告和处理中状态

## Assumptions

- 本次改造仍以桌面 Git 工作流为核心，不扩展新的业务领域
- “参考 IDEA”表示借鉴其交互节奏、信息密度和渐进展开方式，而不是要求像素级复刻
- 产品名称仍可出现在窗口标题、系统层级或关于信息中，但不应在仓库工作区内长期占据主视觉位置
- 现有 Git 能力不应被删除，只允许重组入口和调整层级
- 精简的首要目标是降低认知负担，而不是单纯减少颜色或控件数量

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 打开仓库后的首屏中，用户可在 3 秒内识别当前仓库与当前分支，且无需扫描超过一个主上下文区域
- **SC-002**: 在默认窗口尺寸下，主工作区的主要可见面积优先用于改动列表和差异内容，持久顶部装饰区不再占据显著视觉比例
- **SC-003**: 80% 的常用 Git 动作可通过一个主上下文入口和一次额外选择完成，而不是依赖多个常驻按钮
- **SC-004**: 在稳定状态下，主工作区中不会同时存在多个重复表达同一语义的持久区域，例如重复的仓库信息、section 信息或下一步提示
- **SC-005**: 用户首次进入新版工作区时，可在 10 秒内找到“当前分支入口”和“改动处理区域”
- **SC-006**: 对提交、暂存、取消暂存、切换分支和刷新等高频操作进行回归时，反馈保持简短明确，不会在操作完成后长期占据主空间
- **SC-007**: 精简后的界面仍然保留全部现有核心 Git 能力的可达性，相关能力在回归验证中均可完成
- **SC-008**: 至少一轮人工走查可以明确指出被移除或收纳的冗余元素，包括产品标题、重复芯片、重复说明区和多余常驻按钮
